## Pipeline Data Flow

```
Input: Customer drawing (PDF/DWG/JPG)

Step 1 вҖ” Drawing Understanding
  Claude Opus 4.7 Vision вҶ’ PartFeatures (JSON)

Step 2 вҖ” RAG Retrieval
  PartFeatures вҶ’ embedding text (Claude Haiku)
  вҶ’ Voyage-3-large embedding
  вҶ’ ChromaDB vector search (top-20)
  вҶ’ metadata filter + Voyage rerank-2
  вҶ’ top-3 RetrievedCase

Step 3 вҖ” Process Planning
  PartFeatures + few-shot XML вҶ’ Claude Opus 4.7
  вҶ’ ProcessPlan (JSON, schema-validated)

Step 4 вҖ” Die Parameter Calculation
  PartFeatures + ProcessPlan + few-shot вҶ’ Claude Opus 4.7
  вҶ’ List[DieParameters] (JSON, schema-validated)

Step 5a вҖ” 3D Model Generation
  DieParameters вҶ’ CADQuery parametric templates
  вҶ’ punch.step, die.step, workpiece.step (per station)
  вҶ’ punch.stl, die.stl (for web viewer)

Step 5b вҖ” 2D Drawing Generation
  DieParameters вҶ’ ezdxf templates
  вҶ’ punch_drawing.dxf, die_drawing.dxf (per station)
  вҶ’ production_drawing.dxf, process_breakdown.dxf

Step 6 вҖ” Verification
  Rule-based engine checks:
  - Dimensional consistency
  - Upset ratios вүӨ 2.3
  - Volume conservation Вұ 3%
  - Material compatibility (die HRC > workpiece)
  - Completeness + interference
  вҶ’ On failure: retry Steps 3-5 (max 2Г—)
```

## Data Architecture (3 Layers)

| Layer | Storage | Purpose |
|-------|---------|---------|
| Source of Truth | MinIO (S3) | Raw DWG/PDF files, never modified |
| RAG Store | ChromaDB | Vector + metadata + full case JSON |
| Few-shot Format | In-memory | XML derived from RAG at query time |

## Key Design Decisions

- **LLM decides parameters, code generates geometry** вҖ” Claude outputs JSON, CADQuery builds 3D
- **No LangChain/LlamaIndex** вҖ” direct Anthropic SDK calls, ~100 lines of RAG
- **Phase 1 = single agentic workflow** вҖ” not multi-agent; simpler and more predictable
- **Every output verified** вҖ” physical + geometric rules; human review always required
- **Confidence is mandatory** вҖ” every inference has high/medium/low; low is prominently flagged
